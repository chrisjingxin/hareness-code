# 架构总览

Harness Code 采用“Node 表现层 + Python Agent 内核”的双进程结构：

```text
OpenTUI / Bun CLI  <== JSON-RPC v2 over stdio (JSONL) ==> Python deepagents 内核
packages/cli                                      packages/agent
                  共享方法与载荷类型
                     packages/protocol
```

- `packages/cli/`：参数解析、Python sidecar 生命周期、React/OpenTUI 渲染与交互。
- `packages/protocol/`：语言无关 JSON Schema、生成的 TypeScript/Python 模型和运行时断言。
- `packages/agent/`：配置加载、deepagents/LangGraph 图、模型 adapter 与 JSON-RPC 服务端。

配置来源、TOML v1 契约和项目配置的可信边界见 [配置架构](配置架构.md)。

## 企业 OpenTUI 依赖

CLI 固定 Bun `1.2.19`、企业 `@opentui/core` / `@opentui/react` `0.4.3`、React `19.2.6` 和 `react-reconciler` `0.33.0`。其中 `@opentui/react` 以 `react-reconciler@^0.33.0` 作为传递依赖；`@types/react-reconciler@0.32.0` 仅为企业当前可用的编译期声明。完整直接/传递依赖契约见 `dependency-versions.json` 与企业环境生成的 `bun.lock`。

Core `0.4.3` 的 native library 由其 `@opentui/core-<platform>-<arch>` optional package 提供。企业源必须镜像当前 runner 对应的包，Bun 负责选择；项目不携带 OpenTUI 源码快照、自建 FFI artifact 或 Zig 构建脚本。

TUI 启动时注册 `packages/cli/src/tui/assets/syntax/` 内随源码分发的 Tree-sitter WASM parser 与查询文件。`<markdown>` 使用这些离线资源显示 Agent 输出与 fenced code；网络不可用或未知语言时保持可读降级，审批和 IPC 不受影响。

Python 服务端 stdout 只能输出换行分隔的 JSON-RPC；日志必须进入 stderr 或结构化事件。不得让 TUI 直接依赖 Agent 内部实现，也不得把业务决策放进 `cli`。协议变更必须同时修改两端和跨进程测试。

## JSON-RPC v2

连接建立后必须先调用 `initialize`，协商 `{major, min_minor, max_minor}` 和 capability。首版支持 `run.start`、`run.cancel`、只读配置查询与 `shutdown`；不同 thread 可以并发，同一 thread 只允许一个活动 run。

Agent 输出统一包装为 `event` notification。每个信封包含 `event_id/type/thread_id/run_id/sequence/timestamp_ms/payload`，TUI 以运行内 sequence 拒绝重复和倒序帧，并对序号缺口给出诊断。审批和问答使用 Agent→Client 的标准 JSON-RPC `request`，客户端直接返回 response；不再使用专门的 `respond` 方法。

协议唯一来源为 `packages/protocol/schema/v2.json`。运行：

```bash
bun run protocol:generate
bun run protocol:check
```

生成的 `src/generated.ts` 与 `harness_agent/protocol_generated.py` 禁止手改。两端共同消费 `packages/protocol/fixtures/v2-contract.json`，确保严格字段和兼容行为一致。stdio 单帧上限为 8 MiB，工具参数、结果和审批详情上限为 1 MiB；Python stdout 仍只能输出 JSONL。

v2 直接替换未正式发布的内部 v1，不维护双栈。未来 replay、steer、外部工具和 hooks 必须通过 minor 版本与 capability 增量提供，禁止提前声明未实现能力。

当前协议与运行行为说明保存在 [实施状态](../../.agent/2026-07-14-za38-cli-implementation-status.md)；它是实施交接材料，不是开发者文档入口。

## 执行后端与安全边界

默认 `tools.sandbox = false`，Python sidecar 使用本机 `LocalShellBackend`。TUI 必须清晰显示“本机执行 · 未隔离”。

本机模式在 `create_harness_agent()` 的主 Agent 和默认 `general-purpose` 子 Agent 中分别注册 `WorkspaceBoundaryMiddleware`。它在内置文件工具进入 backend 前执行以下规则：

- `ls`、`read_file`、`write_file`、`edit_file`（以及未来的 `delete`）只接受工作区内的绝对路径。
- `glob`、`grep` 未指定 `path` 时从 backend 的 `root_dir` 即工作区搜索；指定时必须为工作区内的绝对路径。`glob.pattern` 和 `grep.glob` 禁止绝对路径和 `..` 路径段。
- 每个路径均通过 `Path.resolve(strict=False)` canonicalize 后再调用 `relative_to(workspace)`；因此现有父级符号链接、待创建目标的父级符号链接与链接环均不能逃逸。
- 拒绝返回 `ToolMessage(status="error")`，不会调用底层工具，也不会触发 HITL 审批。`WorkspaceBoundaryMiddleware` 不是 shell 沙箱，`execute`、MCP 和自定义工具不在其范围内。

不能只在主 Agent 注册此中间件：deepagents 自动的 `general-purpose` 子 Agent 有独立 middleware 栈。Harness Code 显式传入带有新 guard 实例的默认子 Agent，防止通过 `task` 绕过本机文件边界。MemoryMiddleware 仍通过同一个 backend 在启动时读取 `~/.harness/AGENTS.md`，但以只读提示词注入，不能受该文件工具边界影响或被 Agent 回写。

用户显式设置 `execution.backend = "remote"`、`HARNESS_SANDBOX=true` 或传入 `--sandbox` 时，服务端只接受由用户级或显式配置声明的 `execution.remote.factory`。该工厂必须返回 deepagents `SandboxBackendProtocol`，接收宿主工作区、provider 名称、逻辑远端目录和非秘密 `execution.remote.params`，负责企业认证、工作区同步、网络白名单和远端生命周期；失败时抛出配置错误，禁止回退到本机 backend。项目 TOML 在可信机制完成前会被整体拒绝，之后也不得自动导入远端 provider。

远端模式下 Agent 使用 provider 给出的逻辑工作目录，禁用宿主机 memory 和 skills，避免它们绕开远端执行边界。`js_eval`/QuickJS 已从 Agent 依赖和工具面移除。当前不实现 Docker、Podman 或本机容器 backend。未来接入企业 provider 时，必须补充同步、回写、取消、审计与网络策略的端到端测试。

## 审批模式

`harness_agent.approval_mode` 是唯一的模式解析入口，规范值为 `plan`、`default`、`auto-edit`、`yolo`。它把旧 `ask` 归一为 `default`；未知值不抛出启动错误，而是安全降级并在 `ExecutionSettings.redacted()` 的 `security.approval_mode_warning` 中返回提示。

解析后的值沿 `load_config()` → `JsonRpcServer._create_default_agent()` → `create_harness_agent()` 传递。`approval_policy` 集中提供以下策略，禁止由 TUI 或系统提示词自行解释权限：

- `PlanModeMiddleware` 以明确白名单在工具执行前返回 `ToolMessage`，未知工具默认拒绝；主 Agent 与默认 `general-purpose` 子 Agent 都注册该 guard。
- `WorkspaceBoundaryMiddleware` 仍在实际执行前强制校验；同一策略还作为 HITL 的 `when` 预检复用，因此越界文件调用不会先产生无意义的审批请求。
- `default` 拦截 `execute`、`write_file`、`edit_file`、`delete`、`task`；`auto-edit` 只移除写入/编辑拦截；`yolo` 不注册 Harness HITL。`compact_conversation` 是内部会话维护，不进入 HITL。

CLI 在交互模式才协商 `interactive.approval` 与 `interactive.question`。无头 `-n` 模式不声明这些 capability，服务端遇到审批或提问会立即按拒绝/取消恢复，不发送反向 JSON-RPC request。该 fail-closed 行为是协议层的安全兜底，不应依赖表现层。
