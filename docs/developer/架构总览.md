# 架构总览

Harness Code 采用“Node 表现层 + Python Agent 内核”的双进程结构：

```text
OpenTUI / Bun CLI  <== JSON-RPC v2 over stdio (JSONL) ==> Python deepagents 内核
packages/cli                                      packages/agent
                  共享方法与载荷类型
                     packages/protocol
```

- `packages/cli/`：参数解析、Python sidecar 生命周期、React/OpenTUI 渲染与交互。
- `packages/protocol/`：语言无关 JSON Schema、生成的 TypeScript/Python 模型和运行时断言。
- `packages/agent/`：配置加载、deepagents/LangGraph 图、模型 adapter 与 JSON-RPC 服务端。

配置来源、TOML v1 契约和项目配置的可信边界见 [配置架构](配置架构.md)。

## 企业 OpenTUI 依赖

CLI 固定 Bun `1.2.19`、企业 `@opentui/core` / `@opentui/react` `0.4.3`、React `19.2.6` 和 `react-reconciler` `0.33.0`。其中 `@opentui/react` 以 `react-reconciler@^0.33.0` 作为传递依赖；`@types/react-reconciler@0.32.0` 仅为企业当前可用的编译期声明。完整直接/传递依赖契约见 `dependency-versions.json` 与企业环境生成的 `bun.lock`。

Core `0.4.3` 的 native library 由其 `@opentui/core-<platform>-<arch>` optional package 提供。企业源必须镜像当前 runner 对应的包，Bun 负责选择；项目不携带 OpenTUI 源码快照、自建 FFI artifact 或 Zig 构建脚本。

TUI 启动时注册 `packages/cli/src/tui/assets/syntax/` 内随源码分发的 Tree-sitter WASM parser 与查询文件。`<markdown>` 使用这些离线资源显示 Agent 输出与 fenced code；网络不可用或未知语言时保持可读降级，审批和 IPC 不受影响。

Python 服务端 stdout 只能输出换行分隔的 JSON-RPC；日志必须进入 stderr 或结构化事件。不得让 TUI 直接依赖 Agent 内部实现，也不得把业务决策放进 `cli`。协议变更必须同时修改两端和跨进程测试。

## JSON-RPC v2

连接建立后必须先调用 `initialize`，协商 `{major, min_minor, max_minor}` 和 capability。当前 minor 为 5，支持 `run.start`、`run.cancel`、`context.compact`、配置查询、`models.list`、`threads.list`、`threads.open`、`skills.*` 管理方法与 `shutdown`；不同 thread 可以并发，同一 thread 只允许一个活动 run。`models.list` 要求协商 `models.read`，只返回脱敏 Profile DTO 与可选 Thread 绑定摘要；`run.start.model_profile` 只允许在未绑定 Thread 首次运行时设置 executor。`context.compact` 要求协商 `context.manage`，仅能在空闲 thread 上调用；旧的 v2 客户端仍可协商到其支持的 minor。

Agent 输出统一包装为 `event` notification。每个信封包含 `event_id/type/thread_id/run_id/sequence/timestamp_ms/payload`，TUI 以运行内 sequence 拒绝重复和倒序帧，并对序号缺口给出诊断。`tool_call_id` 是单次工具调用的稳定身份；LangChain 流式 `index` 只用于同一次调用缺失 ID 的续片关联，每个带真实 ID 的新分片都必须更新该临时映射。TUI 以 `run_id + tool_call_id` 保存参数与结果，确保跨轮或跨 run 的同名调用不会覆盖历史。Skill 显式调用会先发送 `skill.loaded`，其正文不会通过事件传输。审批和问答使用 Agent→Client 的标准 JSON-RPC `request`，客户端直接返回 response；不再使用专门的 `respond` 方法。

## TUI Slash Command Registry

交互式命令由 `packages/cli/src/tui/commands.ts` 的不可变 `CommandRegistry` 统一声明。每个 CommandDefinition 包含稳定 ID、canonical 名称、别名、来源、展示类型、参数提示、capability/thread/idle requirements 和安全元数据；帮助、补全、菜单、解析与未知命令建议都只能从该 Registry 导出。Builtin 不可被同名或别名覆盖，后续 Skill、MCP、Plugin Loader 只能构造新的 catalog snapshot。

握手的 `enabled_capabilities` 被收敛为 TUI Runtime 状态。缺少 capability 的命令从菜单隐藏；需要 thread 或空闲状态的命令保留在菜单中但显示禁用原因，实际执行时仍按同一策略复核，防止状态在选择后发生变化。当前 `/continue` 是 `/resume` 的别名，解析后统一为 canonical `resume`；未知 `/command` 只显示本地建议，绝不作为 Prompt 提交。用户需要提交 `/` 开头的普通文本时以 `//` 转义。

ZC-063 负责声明、解析与可用性策略；ZC-064 的 `command-dispatcher.ts` 已将现有 Builtin Handler 迁出根组件。Handler 只按稳定 ID 返回 `notice`、`exit`、本地 Action、Picker、Dialog、RPC 或 Prompt 等 `CommandResult`，`app.tsx` 只适配这些结果并保留 IPC、React 状态和退出生命周期。`/new` 是新建 thread 的 canonical 命令，`/clear` 为兼容别名；活动 run 下通过确认 Dialog 后才取消，`/force-clear` 仅返回迁移提示。

## Project、Thread 与 Message

Harness 的持久化领域模型只有 `project`、`thread` 与 `message`。所有跨层身份字段统一为 `thread_id`；不建立 Harness 的 `session` 概念。LangGraph 原生要求的 `configurable.thread_id` 直接复用该字段。

可恢复状态位于一个用户级 SQLite 文件 `~/.harness/threads.sqlite3`，而不是每个 project 或 thread 各自一个数据库。目录权限固定为 `0700`，新建数据库固定为 `0600`。原始 project 路径不写入数据库：服务端对规范化路径计算 SHA-256 `project_fingerprint`，并同时用于 `harness_threads` 索引筛选和 LangGraph checkpoint 的 `checkpoint_ns`：

```python
{
  "configurable": {
    "thread_id": thread_id,
    "checkpoint_ns": project_fingerprint,
  }
}
```

Harness 自有表为 `harness_threads(project_fingerprint, thread_id, created_at_ms, updated_at_ms, first_message, latest_message, message_count)`，联合主键是 `(project_fingerprint, thread_id)`。LangGraph 维护其 checkpoint 表；`harness_threads` 只索引恢复选择器所需的摘要。`threads.list` 与 `threads.open` 仅在协商 `threads.read` 后可调用，只返回当前 project；JSON-RPC 客户端可以传递内部 `thread_id`，但 TUI 不得将其渲染或要求用户输入。

`harness --resume` 与 `/resume` 共用 TUI 浮层选择器。恢复时先校验索引与 checkpoint，再原子替换 TUI 的历史、交互状态和 sequence；活动 run 或待处理交互期间禁止切换。sidecar 关闭时提交并关闭 SQLite 连接，用户只能在 Harness 停止后手动删除主数据库及其 `-wal`、`-shm` 文件。

## Runtime Profile 与运行状态

Runtime Profile 是可共享 Agent Runtime 的稳定、脱敏配置身份；它不是 thread
或用户会话。v1 Profile Key 以 canonical JSON 的 SHA-256 计算，固定包含
`project_fingerprint`、topology ID/版本、模型角色配置指纹、工具/Skill catalog
指纹、MCP、Sandbox、Policy、middleware 和 Prompt 模板指纹。Key 记录中不保存
原始工作区路径、模型 endpoint、Header、凭据或完整 Prompt。

以下内容不参与 Profile Key，且必须保持在 ThreadStore/Checkpointer 或单次 Run
Context：`thread_id`、消息、PromptEpoch 正文、上下文摘要与 artifact、Todo、审批
状态、取消令牌、run ID 以及未来的 Agent mailbox。相同 Profile 的 thread 因此可在
已由 RuntimePool 复用一张图，而稳定能力变化会构造新 Profile。

`ModelCatalog` 保存命名的 OpenAI-compatible Profile、默认 Profile 和 `planner`、`executor`、`reviewer`、`tester`、`summarizer` 角色绑定；`ModelRouter` 在新 Thread 首次运行时冻结完整脱敏快照。当前 Single Agent topology 只将 `primary ← executor` 纳入 Runtime Profile 并构建模型；ZC-060 的多 Agent topology 会消费同一快照中的其他角色。`ProviderClientPool` 仅按 endpoint/timeout 复用不携带认证信息的 HTTP transport；每个 ChatOpenAI adapter 仍独立保存模型、认证和 Header。

sidecar 以 Runtime Profile 为键从 RuntimePool 租用默认 Agent 图，而不再缓存
`thread_id -> Agent` 或 `thread_id -> ContextWindowMiddleware`。`RunContext` 是
一次 `astream`/`ainvoke` 调用的不可持久化输入，携带 `thread_id`、`run_id`、
PromptEpoch、审批模式、取消令牌、可选 Profile Key 和 Agent 角色。图内
`PromptEpochMiddleware` 在模型调用边界读取它并注入前缀；`ContextWindowMiddleware`
同样优先由它路由到目标 thread。若 LangGraph 的 `configurable.thread_id` 与
RunContext 不一致，调用会 fail closed。

`/.harness/history` 不是图的固定 backend 成员。共享图在工具调用时按当前
RunContext 生成只读虚拟挂载，确保归档只能被所属 thread 读取。手动
`context.compact` 在短期 Runtime lease 内使用对应 Profile 的无 thread 私有成员压缩器，并以请求
中的 thread_id 读写 checkpoint；消息、PromptEpoch、归档、审批恢复和取消状态仍由
ThreadStore、Checkpointer 和 ActiveRun 管理。

`agent_runtime.RuntimePool` 已提供 Profile 级的核心生命周期：相同 Profile Key 的
并发 acquire 共享一个 single-flight 构建任务；`AgentRuntimeLease` 和
`AgentRuntimeRunLease` 分别保护租约、排队 run 与活动 run；状态严格经过
`MISSING → BUILDING → READY → ACTIVE/IDLE → DRAINING → CLOSED`。DRAINING
Runtime 拒绝新 lease，但既有 lease 可以完成或取消；只有 lease/run 均释放后才可
关闭。Runtime 的 `aclose()` 按“取消后台任务 → 刷新持久状态 → Tool → MCP →
Sandbox → Runtime 独占模型资源 → 释放图引用”继续执行，单项失败只记录并不会阻断
后续释放。当前本地实现可使用空资源 Bundle，进程级 Provider client 不属于 Runtime。

sidecar 的 run 与 manual compact 均通过 Pool acquire/release；run 同时持有
`AgentRuntimeRunLease`。Pool 使用 `max_profiles`、空闲 TTL 与 LRU 只淘汰无 lease/run/queue
的未固定 IDLE Runtime；稳定配置变化会使前一个默认 Runtime 进入 DRAINING。关闭顺序是
取消/收敛 run、关闭 RuntimePool、再关闭 ThreadStore；单个 Runtime 的关闭等待受
`close_timeout_seconds` 限制。本任务不改变现有 JSON-RPC 请求路径。

`config.show` 额外返回 `runtime_pool_diagnostics`，因此 `harness config show` 是无需新
capability 的本地诊断入口。摘要包含 Pool 大小、状态计数、活跃 lease/run/queue、命中/未命中、
构建和关闭累计耗时、容量拒绝、按原因聚合的淘汰数、关闭失败数以及最近 64 条生命周期事件。
每条 Runtime/事件只使用 12 位 `profile_id`，绝不返回完整 Profile Key、工作区路径、Prompt、
模型 endpoint 或凭据。`memory.estimated_bytes` / `memory.rss_bytes` 目前恒为 `null` 且状态为
`not_collected`：真实内存采样需独立校准，当前 CI 以 Runtime 数及资源关闭断言保证容量上界。

用户级 SQLite schema v4 新增 `harness_runtime_profiles` 和
`harness_thread_runtime_profiles`；v5 新增 `harness_thread_model_bindings`。模型角色快照与首条
消息在同一事务写入，Profile 记录按 project 指纹与 Profile Key 去重，thread 首次绑定后不可静默替换。
v4 及以前的 thread 不会被推测性绑定到当前模型；升级后以 `legacy immutable binding` 兼容读取，
消息、checkpoint、PromptEpoch 和 context 数据均保持原样。

## Skill 生命周期

Harness 启动时一次性扫描四类来源：内置 `harness_agent/built_in_skills/`、用户 `~/.harness/skills/<name>/`（兼容 `~/.harness/skills/local/<name>/`）、项目 `<workspace>/.harness/skills/<name>/` 和市场安装目录 `~/.harness/skills/market/<market>/<name>/<version>/`。项目 Skill 默认启用，不要求额外信任确认；用户可用 `harness skills disable` 暂停它。来源不会静默覆盖同名项，canonical ID 固定为 `builtin/<name>`、`user/<name>`、`project/<name>` 或 `<market>/<name>`；只有全局唯一时才允许短名称。

每个进程建立带 SHA-256 的不可变 catalog snapshot，运行期间不监听文件变化，以保持提示词 prefix caching 稳定。新 thread 的 PromptEpoch 只注入按 canonical ID 排序、总长最多 4,000 字符的 Skill 元数据（名称、简介、来源、版本和参数提示）；正文、资源路径和运行时错误均不进入 system。Agent 通过统一的 `read_file` 按需读取 `/.harness/skills/<canonical-id>/SKILL.md` 或同根资源，不能调用专用的 `load_skill`/`read_skill_resource` 工具。启用、停用、安装、更新和删除只影响下一新 thread；既有 epoch 与 catalog 视图保持不变。启动后的 `SKILL.md` 篡改、非法 front matter、symlink 和路径穿越都会 fail closed。市场网络和认证不由核心实现，企业包可通过 `harness.skill_marketplaces` entry point 提供 Provider。

## PromptEpoch 与上下文预算

每个默认 Agent thread 在第一次运行时建立并保存不可变 PromptEpoch。其固定装配顺序为：全局核心策略、执行/审批边界、环境快照、只读记忆、Skill 元数据索引、规范化工具 schema；历史摘要、最近轮次和当前用户请求位于该稳定前缀之后。环境快照以输入 SHA-256 缓存 24 小时，恢复 thread 直接从 SQLite 读取 epoch，不重新扫描环境、记忆或 Skill。

模型 Profile 的 `context_window_tokens` 默认 `128000`，输入预算为 `window - clamp(4096, 16384, ceil(window * 10%))`。`ContextWindowMiddleware` 以 UTF-8 字节、消息和 schema 固定开销估算输入：50% 仅报告，60% 归档旧的大工具结果，80% 生成结构化摘要，90% 强制只保留最近一轮。工具调用与结果不会跨 user turn 被拆开。旧工具结果以首尾预览和 artifact ID 留在历史，原文仅写入本机 SQLite，并可由 `/.harness/history/<artifact-id>.md` 分页读取。空摘要、节省不足 20% 或失败均不改写历史；连续三次失败打开熔断器。网关溢出时先归档工具输出，再只允许一次摘要恢复重试。交互客户端可通过 `/compact` 调用 `context.compact` 强制生成一次摘要；它不受自动熔断器限制，但仍要求保存至少 20%，并保留最近两个完整 user turn。

服务端不发送厂商专用缓存字段。它记录 system/schema/历史重写指纹和动态预算；网关未报告缓存 usage 时状态为 `unknown`。协议的 `context.updated` 与 `run.completed.context` 把紧凑状态交给 TUI。

协议唯一来源为 `packages/protocol/schema/v2.json`。运行：

```bash
bun run protocol:generate
bun run protocol:check
```

生成的 `src/generated.ts` 与 `harness_agent/protocol_generated.py` 禁止手改。两端共同消费 `packages/protocol/fixtures/v2-contract.json`，确保严格字段和兼容行为一致。stdio 单帧上限为 8 MiB，工具参数、结果和审批详情上限为 1 MiB；Python stdout 仍只能输出 JSONL。

v2 直接替换未正式发布的内部 v1，不维护双栈。未来 replay、steer、外部工具和 hooks 必须通过 minor 版本与 capability 增量提供，禁止提前声明未实现能力。

当前协议与运行行为说明保存在 [实施状态](../../.agent/2026-07-14-za38-cli-implementation-status.md)；它是实施交接材料，不是开发者文档入口。

## 执行后端与安全边界

默认 `tools.sandbox = false`，Python sidecar 使用本机 `LocalShellBackend`。TUI 必须清晰显示“本机执行 · 未隔离”。

本机模式在 `create_harness_agent()` 的主 Agent 和默认 `general-purpose` 子 Agent 中分别注册 `WorkspaceBoundaryMiddleware`。它在内置文件工具进入 backend 前执行以下规则：

- `ls`、`read_file`、`write_file`、`edit_file`（以及未来的 `delete`）只接受工作区内的绝对路径；唯一例外是 `read_file` 可读取受语法和 thread/project 归属双重校验的 `/.harness/` 虚拟路径。
- `glob`、`grep` 未指定 `path` 时从 backend 的 `root_dir` 即工作区搜索；指定时必须为工作区内的绝对路径。`glob.pattern` 和 `grep.glob` 禁止绝对路径和 `..` 路径段；它们以及 `ls`、写工具和 `execute` 都拒绝 `/.harness/`。
- 每个路径均通过 `Path.resolve(strict=False)` canonicalize 后再调用 `relative_to(workspace)`；因此现有父级符号链接、待创建目标的父级符号链接与链接环均不能逃逸。
- 拒绝返回 `ToolMessage(status="error")`，不会调用底层工具，也不会触发 HITL 审批。`WorkspaceBoundaryMiddleware` 不是 shell 沙箱，`execute`、MCP 和自定义工具不在其范围内。

不能只在主 Agent 注册此中间件：deepagents 自动的 `general-purpose` 子 Agent 有独立 middleware 栈。Harness Code 显式传入带有新 guard 实例的默认子 Agent，防止通过 `task` 绕过本机文件边界。`AGENTS.md` 在新 PromptEpoch 创建时读取一次并作为只读、不可信参考写入 epoch；恢复时不再读取宿主文件。

用户显式设置 `execution.backend = "remote"`、`HARNESS_SANDBOX=true` 或传入 `--sandbox` 时，服务端只接受由用户级或显式配置声明的 `execution.remote.factory`。该工厂必须返回 deepagents `SandboxBackendProtocol`，接收宿主工作区、provider 名称、逻辑远端目录和非秘密 `execution.remote.params`，负责企业认证、工作区同步、网络白名单和远端生命周期；失败时抛出配置错误，禁止回退到本机 backend。项目 TOML 在可信机制完成前会被整体拒绝，之后也不得自动导入远端 provider。

远端模式下 Agent 使用 provider 给出的逻辑工作目录，禁用宿主机 memory 和 skills，避免它们绕开远端执行边界。`js_eval`/QuickJS 已从 Agent 依赖和工具面移除。当前不实现 Docker、Podman 或本机容器 backend。未来接入企业 provider 时，必须补充同步、回写、取消、审计与网络策略的端到端测试。

## 审批模式

`harness_agent.approval_mode` 是唯一的模式解析入口，规范值为 `plan`、`default`、`auto-edit`、`yolo`。它把旧 `ask` 归一为 `default`；未知值不抛出启动错误，而是安全降级并在 `ExecutionSettings.redacted()` 的 `security.approval_mode_warning` 中返回提示。

解析后的值沿 `load_config()` → `JsonRpcServer._create_default_agent()` → `create_harness_agent()` 传递。`approval_policy` 集中提供以下策略，禁止由 TUI 或系统提示词自行解释权限：

- `PlanModeMiddleware` 以明确白名单在工具执行前返回 `ToolMessage`，未知工具默认拒绝；主 Agent 与默认 `general-purpose` 子 Agent 都注册该 guard。
- `WorkspaceBoundaryMiddleware` 仍在实际执行前强制校验；同一策略还作为 HITL 的 `when` 预检复用，因此越界文件调用不会先产生无意义的审批请求。
- `default` 拦截 `execute`、`write_file`、`edit_file`、`delete`、`task`；`auto-edit` 只移除写入/编辑拦截；`yolo` 不注册 Harness HITL。上下文归档和压缩由中间件自动维护，不暴露模型可调用的压缩工具，也不进入 HITL。

可选的 `ShellAllowListMiddleware` 只接受由 POSIX `shlex` 可可靠解析的单一命令。词法层禁止控制运算符、管道、重定向、子 Shell、命令替换、反引号、换行、NUL 与畸形引号；解析失败一律 fail closed。参数可以作为获准程序的普通参数传递，但中间件不解释程序自身语义。绝对路径必须与白名单条目完整匹配，禁止通过 basename 让工作区或临时目录中的同名程序冒充获准程序。同步与异步工具入口复用同一决策函数，并在拒绝时直接返回错误 `ToolMessage`，不调用 backend。

CLI 在交互模式才协商 `interactive.approval` 与 `interactive.question`。无头 `-n` 模式不声明这些 capability，服务端遇到审批或提问会立即按拒绝/取消恢复，不发送反向 JSON-RPC request。该 fail-closed 行为是协议层的安全兜底，不应依赖表现层。
