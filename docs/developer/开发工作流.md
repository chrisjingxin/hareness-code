# 开发工作流

仓库 Markdown 是协作事实来源。每个功能遵循以下闭环：

1. 从 [任务看板](任务看板.md) 找到任务，再通过 `bun run task:claim -- <ID> --owner <名称> --branch <分支>` 认领。
2. 在任务范围内实现并补回归测试；不要顺带重构无关代码。
3. 用户可感知变更更新 `docs/user/`；架构、协议或配置变更更新 `docs/developer/`。
4. 记录测试证据，执行 `bun run task:complete -- <ID> --evidence "..." --references "..."`。
5. 如有版本影响，仅通过 `bun run version:set <SemVer>` 更新版本与 Changelog。

交互式开发必须在仓库根目录直接运行 `bun run dev -- --cwd <工作区>`。根 `dev` 命令会直接启动 CLI，以保留真实终端的 stdin/stdout；不要通过会捕获输出的任务运行器再转发它。

提交前至少运行：

```bash
bun run project:check
bun run typecheck
bun run test
```

构建相关改动再运行 `bun run build`。`.agent/` 只保存 Agent 的计划、实现状态和交接，不替代本文档、任务源或用户文档。

涉及 OpenTUI 时，先确认企业源同时提供 `@opentui/core` / `@opentui/react` `0.4.3` 及当前 runner 的 Core native optional package。随后运行 `bun install --frozen-lockfile` 与 `bun run test:ts`；React `testRender` 会加载当前目标 FFI、创建测试 renderer 并捕获 80×24 与 130×40 帧。Python 依赖改动使用 `cd packages/agent && uv lock && uv sync --extra test`，企业索引和凭据只通过环境变量或 CI 注入。

详细要求见 [变更检查清单](变更检查清单.md) 和 [仓库协作规范](../../AGENTS.md)。
