# 配置架构

Harness Code 使用 TOML 作为用户可维护的配置格式。配置文件只描述非秘密行为；API Key、Token、认证 Header 和企业凭据必须通过明确的 `*_env` 字段引用环境变量。

## v1 契约

每个生效的 TOML 文件必须包含：

```toml
[config]
version = 1
```

当前已实现区段为 `[models]`、`[approval]` 和 `[execution]`。模型必须定义一个 `default_profile`，当前内核只执行该单一 Profile；多 Profile 选择由 [ZC-019](tasks/ZC-019.md) 实现。每个 `[models.profiles.<name>]` 可设置 `context_window_tokens`；默认 `128000`，显式值最小 `16384`，并通过 `config show` 的 `context_window_source` 说明来自默认还是 TOML。

用户模板位于 [config.toml](../user/examples/config.toml)。它是注释完整的长期配置表面，不是工作区配置样例。

## 来源与优先级

长期优先级为：managed 策略锁 > CLI 参数 > `HARNESS_*` 环境变量 > `--config` > 已信任项目本地 > 已信任项目共享 > 用户配置 > 默认值。managed 策略无论来源顺序都不能被下层放宽。

| 来源 | 当前状态 | 可配置范围 |
| --- | --- | --- |
| 系统 managed TOML | 计划中，[ZC-038](tasks/ZC-038.md) | 企业不可覆盖策略 |
| CLI 参数 | 已实现 | 当前仅 `--sandbox`，高于普通环境变量 |
| `HARNESS_*` 环境变量 | 已实现 | 模型、审批和执行临时覆盖 |
| `--config <PATH>` | 已实现 | 用户明确选择的可信 TOML |
| `~/.harness/config.toml` | 已实现 | 用户模型、审批和执行设置 |
| `.harness/config.local.toml` | 计划中，[ZC-037](tasks/ZC-037.md) | 仅可信项目的受限本地偏好 |
| `.harness/config.toml` | 计划中，[ZC-037](tasks/ZC-037.md) | 仅可信项目的受限共享设置 |

当前未实现项目可信机制，因此只要工作区内出现上述任一项目配置，启动会在模型初始化前失败。用户必须把配置移至用户目录，或显式传入该文件的 `--config` 路径。显式传入代表用户已确认该文件来源。当前 `.gitignore` 继续忽略这两个项目路径，ZC-037 实现可信共享配置时再调整 Git 追踪策略。

## 安全规则

- 不自动加载项目 `.env`，也不支持 `$VAR` 或 `${VAR}` 的通用插值。
- `api_key_env` 和 `headers_env` 的值必须是大写环境变量名；模型密钥不写入 TOML、JSON-RPC、日志或 TUI。
- 固定 Header 只能保存非秘密值。`Authorization`、`Cookie`、`X-Api-Key` 等认证 Header 必须使用 `headers_env`。
- 远端 sandbox factory、MCP、Hook、Extension 和 Plugin 都不得因为克隆项目自动导入、下载或执行。
- 后续 managed TOML 路径固定为 macOS `/Library/Application Support/Harness/managed.toml`、Linux/WSL `/etc/harness/managed.toml`、Windows `%ProgramData%\\Harness\\managed.toml`；企业分发机制不在当前阶段实现。

## 维护方式

`packages/agent/harness_agent/config_manifest.py` 是字段状态、来源许可和通用安全规则的唯一实现来源。增加任一配置字段时必须同步更新 Manifest、解析器、模板、用户文档、本文档和对应任务；未实现字段只能以注释形式出现在模板中。
