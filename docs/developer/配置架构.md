# 配置架构

Harness Code 使用 TOML 作为用户可维护的配置格式。环境变量仍是秘密的默认来源；唯一字面量例外是用户级 `~/.harness/config.toml` 模型 Profile 中的 `api_key` 降级值。Token、认证 Header、企业凭据以及其他来源的 API Key 仍必须通过明确的 `*_env` 字段引用环境变量。

## v1 契约

每个生效的 TOML 文件必须包含：

```toml
[config]
version = 1
```

当前已实现区段为 `[models]`、`[approval]`、`[execution]` 和 `[runtime_pool]`。`[models]` 包含多个命名的 OpenAI-compatible Profile、`default_profile` 以及可选 `[models.roles]`。固定 `provider = "openai-compatible"`；`provider_label` 仅用于脱敏 TUI 展示。角色只允许 `planner`、`executor`、`reviewer`、`tester`、`summarizer`，未绑定角色回退到默认 Profile。每个 `[models.profiles.<name>]` 可设置 `context_window_tokens`；默认 `128000`，显式值最小 `16384`，并通过 `config show` 的 `context_window_source` 说明来自默认还是 TOML。能力未声明时兼容为 `tool-calling`、`streaming`；显式能力必须来自 `tool-calling`、`streaming`、`vision`。

新 Thread 的首次 `run.start` 会在记录首条消息的同一 SQLite 事务中冻结脱敏的角色/Profile 快照。快照只含 Profile ID、Provider 标签、模型名、能力与窗口，绝不含 endpoint、Header、API Key 或环境变量值。当前 Single Agent Runtime 仅纳入 `primary ← executor` 的模型指纹；未来 topology 才将实际使用的额外角色纳入同一 Runtime Profile Key。模型名称、endpoint、超时、重试、窗口、能力、固定 Header 名称/值和环境变量名只参与哈希；数据库和诊断只保存不可逆组件指纹，绝不保存 API Key、认证 Header、原始工作区路径或完整提示词。`thread_id`、消息、审批、PromptEpoch 正文和 run ID 不属于配置身份，不能进入该 Key。

用户模板位于 [config.toml](../user/examples/config.toml)。它是注释完整的长期配置表面，不是工作区配置样例。

## 来源与优先级

长期优先级为：managed 策略锁 > CLI 参数 > `HARNESS_*` 环境变量 > `--config` > 已信任项目本地 > 已信任项目共享 > 用户配置 > 默认值。managed 策略无论来源顺序都不能被下层放宽。

| 来源 | 当前状态 | 可配置范围 |
| --- | --- | --- |
| 系统 managed TOML | 计划中，[ZC-038](tasks/ZC-038.md) | 企业不可覆盖策略 |
| CLI 参数 | 已实现 | 当前仅 `--sandbox`，高于普通环境变量 |
| `HARNESS_*` 环境变量 | 已实现 | 模型、审批和执行临时覆盖 |
| `--config <PATH>` | 已实现 | 用户明确选择的可信 TOML |
| `~/.harness/config.toml` | 已实现 | 用户模型、审批和执行设置 |
| `.harness/config.local.toml` | 计划中，[ZC-037](tasks/ZC-037.md) | 仅可信项目的受限本地偏好 |
| `.harness/config.toml` | 计划中，[ZC-037](tasks/ZC-037.md) | 仅可信项目的受限共享设置 |

当前未实现项目可信机制，因此只要工作区内出现上述任一项目配置，启动会在模型初始化前失败。用户必须把配置移至用户目录，或显式传入该文件的 `--config` 路径。显式传入代表用户已确认该文件来源。当前 `.gitignore` 继续忽略这两个项目路径，ZC-037 实现可信共享配置时再调整 Git 追踪策略。

## 安全规则

- 不自动加载项目 `.env`，也不支持 `$VAR` 或 `${VAR}` 的通用插值。
- `api_key_env` 和 `headers_env` 的值必须是大写环境变量名。非空 `api_key_env` 始终优先；只有用户级 TOML 可提供非空 `api_key` 降级值。
- Unix 上只要用户 TOML 存在 `api_key`，文件就不得存在 group/other 权限；该规则在环境变量覆盖密钥时也会执行。Windows 不使用 POSIX mode 校验 ACL。
- 最终密钥不进入 JSON-RPC、日志、TUI、提示词或本地 Shell 环境；脱敏摘要只显示 `environment` / `toml` / `missing` 来源。
- 固定 Header 只能保存非秘密值。`Authorization`、`Cookie`、`X-Api-Key` 等认证 Header 必须使用 `headers_env`。
- 远端 sandbox factory、MCP、Hook、Extension 和 Plugin 都不得因为克隆项目自动导入、下载或执行。
- 后续 managed TOML 路径固定为 macOS `/Library/Application Support/Harness/managed.toml`、Linux/WSL `/etc/harness/managed.toml`、Windows `%ProgramData%\\Harness\\managed.toml`；企业分发机制不在当前阶段实现。

## 维护方式

`packages/agent/harness_agent/config_manifest.py` 是字段状态、来源许可和通用安全规则的唯一实现来源。增加任一配置字段时必须同步更新 Manifest、解析器、模板、用户文档、本文档和对应任务；未实现字段只能以注释形式出现在模板中。

## RuntimePool 边界

`[runtime_pool]` 仅配置 sidecar 内存中可重建的 Runtime，不改变 thread 持久化语义。允许的字段及范围为：`max_profiles = 1..64`（默认 `8`）、`idle_ttl_seconds = 60..86400`（默认 `1800`）、`close_timeout_seconds = 1..120`（默认 `15`）与 `pin_default_profile`（默认 `false`）。配置不接受环境变量或 CLI 覆盖，避免瞬态进程环境意外放大资源占用。

Pool 到达容量时，只会按 LRU 淘汰 `IDLE`、无 lease/run/queue 且未固定的 Runtime；TTL sweep 使用相同条件。没有安全候选时返回 `RUNTIME_POOL_CAPACITY_EXHAUSTED`，禁止以临时 Runtime 绕过上限。稳定能力配置变化会创建新 Profile，并将前一个默认 Runtime 标记为 `DRAINING`：它拒绝新 lease，但允许已有 run 完成，最后一个引用释放后才关闭。sidecar 退出的顺序固定为“停止/取消 run → 关闭 RuntimePool（单 Runtime 受 close timeout 限制）→ 关闭 ThreadStore”。
