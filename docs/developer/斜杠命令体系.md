# Slash Command 体系实施设计

## 定位与边界

Slash Command 是交互式 Harness TUI 的功能入口，而不是面向脚本的命令行子命令树。它以少量稳定的顶层命令打开 OpenTUI 的 Picker、Manager、Viewer、Dialog 或执行明确的 Action；CLI Subcommand 与 JSON-RPC 则分别服务自动化和跨进程通信，三者复用底层服务但不共享文本语法。

```text
Slash Command → Command Registry → Dispatcher → CommandResult → TUI Adapter
                                      │
                                      └→ Domain Service / JSON-RPC
```

ZC-063 已将 `packages/cli/src/tui/commands.ts` 的静态定义升级为不可变 Command Registry：帮助、补全、菜单、解析、capability 可见性与未知命令建议共享同一份描述。ZC-064 再通过 `command-dispatcher.ts` 将现有 Builtin Handler 迁出 `app.tsx`；Handler 仅返回 `CommandResult`，根组件只适配 RPC、React 状态、Picker/Dialog 和退出副作用。

ZC-065 在 `packages/cli/src/tui/overlays.tsx` 提供两项可复用基础：`SearchPicker<T>` 统一遮罩、尺寸、窄终端降级、搜索焦点、加载/错误/空态、滚动与 Enter 选择；调用方只提供稳定键和领域行渲染。`DialogShell` 复用同一浮层骨架，并接收确认、取消和可选内容。Skill、Thread 等领域组件继续持有过滤、选中和 Shortcut 状态；浮层从可见转为关闭时，在允许输入的状态下延后一 tick 恢复 Composer 焦点。后续 `/model` 只需提供 `SearchPicker<Model>` 的行渲染与选择语义，不能重新实现布局或焦点生命周期。

ZC-067 已以该接口实现 `/model [query]`：命令要求 `models.read` 且在活动 run/交互中禁用，Dispatcher 返回 `open-picker(models, initialQuery)`。Picker 只消费 `models.list` 的脱敏 DTO；确认后仅写入 Composer 的 `pendingModelProfile`，下一次尚未绑定 Thread 的 `run.start` 才携带 `model_profile`。`/new`、成功启动和取消选择会清理该状态。已绑定 Thread 不打开 Picker，而以 `DialogShell` 展示实际绑定或 `legacy immutable binding`，只提供新建 Thread 动作，不允许热切换。

## 交互约定

- Picker：`/model`、`/agent`、`/resume`、`/theme`。参数仅作为初始查询；唯一匹配可以走明确选择流程。
- Manager：`/skills`、`/mcp`、`/tools`、`/permissions`、`/settings`。资源的详情、筛选、启停和确认在 TUI 内完成。
- Viewer：`/context`、`/diff`、`/status`、`/usage`、`/history`、`/doctor`、`/help`。
- Dialog：`/new`、`/compact`、`/rename`、`/fork`、`/rewind`、`/export`、`/reload`。
- Action：`/undo`、`/redo`、`/copy`、`/quit`。

默认菜单只推荐 `/new`、`/resume`、`/model`、`/agent`、`/skills`、`/context`、`/diff`、`/compact`、`/undo`、`/permissions`、`/mcp`、`/settings`、`/status`、`/help` 和 `/quit`。其余命令可搜索但不占用首屏。

`/new` 为创建新 thread 的 canonical 名称，`/clear` 保留为别名；活动 run 下会先打开确认 Dialog，确认后才发起取消。`/force-clear` 仅在一个兼容周期内提示迁移到 `/new`，不会再形成第二套取消/清理流程。`/resume` 的别名为 `/continue` 和 `/threads`。不向用户展示内部 `thread_id`。

## 命令契约

每项命令必须拥有稳定 ID、canonical 名称、别名、来源、类别、展示类型、参数提示、所需 capability、thread/空闲状态要求、安全确认策略和执行器。执行器只返回结构化 `CommandResult`，例如 `open-picker`、`open-manager`、`open-viewer`、`open-dialog`、`rpc`、`notice`、`submit-prompt` 或 `exit`；TUI 根组件只负责将结果映射为界面动作。

Registry 在生成菜单、帮助和补全前统一计算可用性：缺少协商 capability 时隐藏；当前状态不满足时显示禁用原因；执行中的 run 不允许切换 Profile、恢复其他 thread 或进行破坏性操作。命令名称大小写不敏感，参数原文保持不变；未知 `/command` 显示建议而不能发送给模型，`//text` 是向模型发送以 `/` 开头文本的转义方式。

## 状态与安全限制

- Runtime Profile 绑定后的 thread 不能被静默改用其他模型或 AgentTopology。`/model`、`/agent` 在已绑定 thread 上只能展示当前配置，并明确引导用户创建新 thread 或 fork，不能在运行中热切换。
- `/compact [focus]` 始终进入确认 Dialog；`focus` 只作为压缩提示输入，压缩仍遵守既有最近轮次、节省率、归档和隔离规则。
- `/undo`、`/redo`、`/diff` 只处理 Harness 记录且可验证归属的 Agent 变更，绝不以 `git reset --hard` 处理用户原有脏改动。
- Settings 和 Permission Manager 只能修改经校验的白名单字段，必须尊重 managed lock、项目可信边界和现有审批模式；它们不得写入 API Key、秘密 Header 或绕过审批策略。
- MCP、Tool、Plugin、Skill 与诊断视图均不得显示凭据、完整敏感 URL、Prompt、原始大工具结果或内部 mailbox 内容。

## 动态来源与重载

Builtin 命令不能被覆盖。Skill、MCP Prompt、Plugin Command 与 Prompt Template 通过 Loader 进入同一 Registry，并显示来源。短名称只在全局唯一时可调用；冲突项必须使用稳定限定名，保留 `/skill:<canonical-id>` 兼容形式。

不启用文件监听。显式 `/reload` 只允许在空闲状态下构建新 catalog；既有 PromptEpoch、活动 run 和已打开资源保持其旧快照，只有之后新建的 thread 使用新 catalog。没有对应 capability 或可信来源时，命令必须隐藏或拒绝。
