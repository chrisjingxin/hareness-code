# 模型配置

v0.1 只支持 OpenAI 兼容协议。TOML 使用 v1 结构；当前只读取用户配置 `~/.harness/config.toml` 和用户明确传入的 `--config <PATH>`。覆盖优先级为 `--sandbox` 等 CLI 参数 > `HARNESS_*` 环境变量 > `--config` > 用户配置 > 默认值。

## 配置文件

从仓库中的 [用户级示例配置](examples/config.toml) 开始：

```toml
[config]
version = 1

[models]
default_profile = "fast"

[models.profiles.fast]
provider = "openai-compatible"
provider_label = "企业 Fast 网关"
model = "your-fast-coding-model"
base_url = "https://gateway.example.internal/v1"
api_key_env = "HARNESS_API_KEY"
context_window_tokens = 128000
capabilities = ["tool-calling", "streaming"]

[models.profiles.pro]
provider = "openai-compatible"
provider_label = "企业 Pro 网关"
model = "your-pro-coding-model"
base_url = "https://gateway.example.internal/v1"
api_key_env = "HARNESS_PRO_API_KEY"
context_window_tokens = 256000
capabilities = ["tool-calling", "streaming", "vision"]

[models.roles]
planner = "pro"
executor = "fast"
```

`api_key_env` 填写环境变量名，而不是密钥值。非空环境变量是首选来源。如果本机开发环境无法注入环境变量，可仅在 `~/.harness/config.toml` 的同一 Profile 中增加：

```toml
api_key = "你的企业网关密钥"
```

这是低安全性的降级方案。macOS/Linux 上 Harness 首次加载含 `api_key` 的用户级配置时会自动将文件权限收紧为 `600`；若文件系统拒绝修改权限，Harness 会拒绝使用该明文密钥并提示改用 `api_key_env`。该字段不得出现在显式 `--config` 或项目 TOML 中，配置文件也不得提交到仓库。

```bash
export HARNESS_API_KEY='你的企业网关密钥'
bun run dev -- config path
bun run dev -- config show
```

`context_window_tokens` 是模型 Profile 的输入窗口，用于上下文预算和压缩。未填写时为 `128000`；显式值不得小于 `16384`。`config show` 会显示最终值与 `context_window_source=default|config`，不依赖网关是否返回模型窗口。

`config show` 仅展示脱敏后的有效配置。`api_key_source` 只会显示 `environment`、`toml` 或 `missing`，绝不返回密钥值。可用环境变量包括 `HARNESS_MODEL`、`HARNESS_BASE_URL`、`HARNESS_API_KEY_ENV`、`HARNESS_TIMEOUT_SECONDS`、`HARNESS_MAX_RETRIES`、`HARNESS_APPROVAL_MODE` 和 `HARNESS_SANDBOX`。`HARNESS_AGENT_PYTHON` 仅用于开发时定位 sidecar，不属于 TOML 配置字段。

每个 Profile 只支持 `provider = "openai-compatible"`。`provider_label` 只用于 TUI 分组和展示，默认是 `OpenAI-compatible`；它不启用 Anthropic、Gemini 等原生 SDK。`capabilities` 可声明 `tool-calling`、`streaming` 和 `vision`；不写时兼容为 `tool-calling`、`streaming`。未知能力、未知角色或引用不存在的 Profile 会阻止加载；缺失凭据会把该 Profile 标记为不可用，不能被选择或执行。Harness 不会静默降级。

企业网关若需要非秘密固定头，可用 `[models.profiles.<name>.headers]`；若头值来自环境变量，使用 `[models.profiles.<name>.headers_env]`。认证 Header、Cookie 和除上述用户级 `api_key` 外的密钥仍只能使用 `headers_env` 或对应的 `*_env` 字段。

`[models.roles]` 可绑定 `planner`、`executor`、`reviewer`、`tester`、`summarizer`。未声明的角色回退到 `default_profile`。当前 Single Agent 仅将 `executor` 解析为实际 `primary` Runtime 模型；`planner` 等绑定会随 Thread 一起冻结，供后续多 Agent topology 使用，但本期不会触发 Planner→Executor 调度。

## 在 TUI 中选择模型

交互式运行可输入 `/model` 或 `/models` 打开模型 Profile Picker；`/model pro` 会将 `pro` 作为初始筛选条件。Picker 仅显示 Profile ID、模型名、Provider 标签、能力、窗口和可用状态，绝不显示 endpoint、Header、API Key 或环境变量值。

选择不会修改 TOML 默认值，也不会热切换既有 Thread。它只保存为 composer 的待选 Profile，并在下一条创建新 Thread 的 `run.start` 中绑定为 `executor`：

- `/new` 会清除待选 Profile，回到 `default_profile`。
- 成功启动新 Thread 后待选 Profile 自动清除。
- 已绑定 Thread 使用 `/model` 时只显示其脱敏绑定摘要，并要求新建 Thread；无法准确识别的旧 Thread 显示 `legacy immutable binding`。
- 运行中、等待审批或问题时 `/model` 禁用；不可用 Profile 会显示原因但不能确认。

工作区 `.harness/config.toml` 和 `.harness/config.local.toml` 当前不会被自动读取。检测到这两个文件时，Harness Code 会在发起模型请求前失败；请移至 `~/.harness/config.toml`，或由你主动传入 `--config <PATH>`。

工具执行默认在本机。企业远端 sandbox 的配置、启用方式和限制见 [安全与沙箱](安全与沙箱.md)。

## RuntimePool

sidecar 会让稳定配置相同的 thread 共享 AgentRuntime；SQLite 中的 thread 历史、PromptEpoch 和 checkpoint 始终独立保存。默认的有界缓存如下，可在用户级或显式 TOML 调整：

```toml
[runtime_pool]
max_profiles = 8
idle_ttl_seconds = 1800
close_timeout_seconds = 15
pin_default_profile = false
```

`max_profiles` 范围为 `1-64`，`idle_ttl_seconds` 为 `60-86400`，`close_timeout_seconds` 为 `1-120`。到达容量或 TTL 后，只会淘汰没有活动 run、租约或排队任务的 Runtime；下次使用对应 thread 会重建运行环境并从 SQLite 恢复状态。一般保持 `pin_default_profile = false`；设为 `true` 会使默认 Profile 不参与 LRU/TTL 淘汰，适合配置固定的单项目长会话。配置产生新的 Profile 时，旧 Profile 仍会进入 DRAINING，以免固定缓存阻止新配置生效。
