# 故障排查

## 找不到模型配置或 API Key

确认 `~/.harness/config.toml` 或 `--config` 指向的文件包含 `[config]`、`[models]` 和默认 Profile，并为 `api_key_env` 指定的环境变量设置密钥。如使用用户 TOML 的 `api_key` 降级值，macOS/Linux 上 Harness 会在首次加载时自动收紧文件权限为 `600`；显式 `--config` 和工作区配置不接受明文密钥。工作区 `.harness/config.toml` 目前会被拒绝。使用以下命令确认生效配置；输出不会包含密钥：

```bash
bun run dev -- config path
bun run dev -- config show
```

`api_key_source=environment|toml|missing` 可用于确认密钥来源，不会显示密钥内容。

## Python sidecar 无法启动

开发模式优先使用 `packages/agent/.venv/bin/python`。确认虚拟环境存在、Python 版本为 3.11+，并在仓库根目录运行命令；首次或依赖迁移后执行 `cd packages/agent && uv sync --extra test`。需要临时指定解释器时设置 `HARNESS_AGENT_PYTHON`。

使用 `--cwd <路径>` 时，该目录必须已经存在；CLI 不会自动创建工作区。显式配置路径建议使用绝对路径，避免它随 Agent 工作目录变化：

```bash
mkdir -p /private/tmp/za38-smoke
bun run dev -- --cwd /private/tmp/za38-smoke \
  --config "$HOME/.harness/config.toml"
```

## 终端界面显示简化

在窄终端（小于 88×28）或 `TERM=dumb` 下，星空与品牌动画会主动关闭以保证可读性。请扩大终端，或使用支持 ANSI 色彩的终端模拟器。

若界面出现 `^[[`、`rgb:`、`Warp(...)` 或鼠标坐标等控制序列，说明 TUI 被管道或输出捕获器启动。请在真实终端的仓库根目录使用 `bun run dev -- ...`，不要通过会捕获 stdout/stderr 的任务运行器启动交互界面。

若启动提示缺少 OpenTUI native artifact，请检查企业源是否已镜像 `@opentui/core@0.4.3` 对应当前平台和架构的 optional package，例如 Apple Silicon 使用 `@opentui/core-darwin-arm64@0.4.3`。项目不再自行编译或回退到不匹配的平台库。

## 网关请求失败

检查 `base_url` 是否为企业 OpenAI 兼容接口根路径、模型名是否可用、企业代理和必要请求头是否已配置。切勿把真实密钥贴进 Issue、终端截图或仓库文件。

## 协议版本或序号错误

CLI 与 Python sidecar 必须来自同一发行版本。出现 `PROTOCOL_MISMATCH` 时，检查是否混用了旧的 Python 环境或 CLI 构建产物；源码开发时重新运行 `bun run build` 并确认 `HARNESS_AGENT_PYTHON` 指向本仓库虚拟环境。

thread 中出现“协议序号缺口”表示同一运行的事件不连续。界面会继续处理后续事件，但应保留 stderr 和终端截图用于诊断。不要把包含密钥、工具完整参数或企业路径的原始帧直接提交到公开渠道。

## 无法恢复 Thread

确认使用的是交互式 `harness --resume` 或 `/resume`，并且当前 project 与创建该 thread 时相同。`THREAD_NOT_RECOVERABLE` 表示索引存在但对应 LangGraph checkpoint 已不存在，重新开启一个 thread 即可；`THREAD_STORE_UNAVAILABLE`、`CHECKPOINT_DATABASE_CORRUPT` 或 `CHECKPOINT_SCHEMA_TOO_NEW` 表示本地数据库不可用或版本不兼容。停止 Harness 后，可备份或删除 `~/.harness/threads.sqlite3`、`-wal` 和 `-shm`，随后重新启动创建新的数据库。

如果问题涉及尚未实现的能力，请先查看 [任务看板](../developer/任务看板.md)；若不是已知待办，请按 [开发工作流](../developer/开发工作流.md) 提供最小复现和日志摘要。

## RuntimePool 容量不足或资源关闭超时

`RUNTIME_POOL_CAPACITY_EXHAUSTED` 表示当前 RuntimePool 已达到 `max_profiles`，而所有已有 Runtime 都正在执行、被租用或固定，sidecar 不会为了接收新请求而中断它们。等待当前任务完成后重试，或在用户级 `[runtime_pool]` 中适度增大 `max_profiles`（最大 `64`）。

如果退出时日志出现 `Runtime close timed out`，说明某个 Runtime 的资源关闭超过 `close_timeout_seconds`。这不会删除 SQLite 中的 thread 数据；重启后可以恢复会话。先检查是否有未响应的工具、Sandbox 或 MCP 连接，再按需将该值提高到最大 `120` 秒。不要把完整 Prompt、模型凭据或企业路径写入诊断日志。

使用以下命令读取当前 sidecar 的脱敏 Pool 摘要：

```bash
bun run dev -- config show
```

输出中的 `runtime_pool_diagnostics` 会显示 Pool 大小、活跃 lease/run、命中/未命中、构建与关闭累计耗时、淘汰原因、关闭失败及最近生命周期事件。`profile_id` 只是短诊断标识，不能用于推导模型、工作区或会话内容；`memory.status = not_collected` 表示当前版本不采集不稳定的进程 RSS，排障应先查看 Runtime 数量、淘汰与关闭结果。
